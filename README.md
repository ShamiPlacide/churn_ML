# Telco Customer Churn — End-to-End MLOps Pipeline

A complete, production-style machine learning pipeline: train a model, track experiments with MLflow, serve predictions through a FastAPI service, containerize it with Docker, and validate every change with automated tests in CI.

Built to demonstrate ML **engineering**, not just model-fitting — the parts of the job that don't show up in a Jupyter notebook.

## Problem

Predict whether a telecom customer is likely to **churn** (cancel their subscription) based on their account and service details, using the [IBM Telco Customer Churn dataset](https://github.com/IBM/telco-customer-churn-on-icp4d) (7,043 customers, 20 features).

## Architecture

```
 data/telco_churn.csv
        │
        ▼
 src/preprocess.py   (shared feature pipeline — used by both training and serving)
        │
        ▼
 src/train.py  ──────────►  MLflow tracking (sqlite) — logs params, metrics, model artifacts
        │                   for every candidate model
        ▼
 models/model.joblib  (best model by F1 score, selected automatically)
        │
        ▼
 app/main.py  (FastAPI)  ──►  /predict, /health
        │
        ▼
 Dockerfile  ──►  containerized service
        │
        ▼
 .github/workflows/ci.yml  ──►  lint → train → test → docker build, on every push
```

**Design choice worth calling out:** preprocessing logic lives in one shared module (`src/preprocess.py`) used by both training and the API. This avoids *train/serve skew* — a common real-world bug where the API transforms incoming data slightly differently than training did, silently degrading predictions.

## Results

Two candidate models are trained and logged to MLflow on every run; the best one by F1 score is automatically selected and saved.

| Model | Accuracy | Precision | Recall | F1 | ROC-AUC |
|---|---|---|---|---|---|
| Logistic Regression | 0.737 | 0.503 | 0.783 | 0.613 | 0.841 |
| **Random Forest (selected)** | **0.758** | **0.531** | **0.767** | **0.627** | **0.842** |

Recall is prioritized via `class_weight="balanced"` — in a churn use case, missing a customer who's about to leave is more costly than a false alarm.

## Running locally

```bash
pip install -r requirements.txt

# Train the model (logs to MLflow, saves the best model to models/model.joblib)
python -m src.train

# Inspect experiment runs
mlflow ui --backend-store-uri sqlite:///mlflow.db

# Serve predictions
uvicorn app.main:app --reload
```

Then hit the API:

```bash
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "gender": "Female", "SeniorCitizen": 0, "Partner": "Yes",
    "Dependents": "No", "tenure": 1, "PhoneService": "No",
    "MultipleLines": "No phone service", "InternetService": "DSL",
    "OnlineSecurity": "No", "OnlineBackup": "Yes",
    "DeviceProtection": "No", "TechSupport": "No",
    "StreamingTV": "No", "StreamingMovies": "No",
    "Contract": "Month-to-month", "PaperlessBilling": "Yes",
    "PaymentMethod": "Electronic check",
    "MonthlyCharges": 29.85, "TotalCharges": 29.85
  }'
```

## Running with Docker

```bash
docker compose up --build
# API available at http://localhost:8000
```

## Testing

```bash
pytest tests/ -v
```

Tests cover: health check, a valid prediction request, and input validation (rejecting out-of-range or malformed fields before they ever reach the model).

## CI/CD

Every push runs a GitHub Actions pipeline that lints with `ruff`, retrains the model, runs the test suite, and builds the Docker image — so a broken change never reaches `main` silently.

## Possible extensions

- Model registry + staged promotion (staging → production) via MLflow Model Registry
- Data drift monitoring on incoming prediction requests
- Feature importance / SHAP explanations exposed through the API
